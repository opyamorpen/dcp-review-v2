## 1. 系统架构

### 1.1 整体架构

DCP 评审中心是一个基于 ONES 插件平台开发的团队级 DCP（Decision Checkpoint）评审管理系统，采用"前端模块 + 后端 API + 实体存储"三层架构：

```
┌──────────────────────────────────────────────────────────────┐
│                        ONES 平台宿主                            │
│  ┌─────────────┐  ┌──────────────┐  ┌─────────────────────┐  │
│  │ 项目内组件    │  │  评审人工作台   │  │   侧边栏应用容器      │  │
│  │dcp-review-tab│  │dcp-reviewer-  │  │   dcp-sidebar       │  │
│  │(ProjectCustom│  │workspace      │  │ ┌─────────────────┐ │  │
│  │ Component)   │  │(ones:workspace│  │ │dcp-template-config│ │  │
│  │              │  │ :new)         │  │ │dcp-review-overview│ │  │
│  └──────┬───────┘  └──────┬────────┘  │ └────────┬────────┘ │  │
│         │                 │           └──────────┼──────────┘  │
│         └──────────┬──────┴──────────────────────┘             │
│                    ▼                                            │
│         ┌──────────────────────────┐                            │
│         │   共享 API 层 (api.ts)     │                            │
│         │ - getTeamUUID / getAppID   │                            │
│         │ - checkPermission          │                            │
│         │ - apiGet / apiPost         │                            │
│         └────────────┬─────────────┘                            │
└──────────────────────┼─────────────────────────────────────────┘
                       │ HTTP (Ones-Plugin-Id 头)
                       ▼
┌──────────────────────────────────────────────────────────────┐
│              后端 External API (backend/src/index.ts)           │
│  ┌──────────┐ ┌──────────┐ ┌──────────┐ ┌──────────────────┐  │
│  │ 配置管理   │ │ 评审单管理 │ │ 材料管理   │ │ 决议/审计/通知      │  │
│  │ Config   │ │ Review   │ │ Material │ │Resolution/Audit  │  │
│  └────┬─────┘ └────┬─────┘ └────┬─────┘ └────┬─────────────┘  │

│         │                            │                          │
│         ▼                            ▼                          │
│   ONES 内部 API (OPFetch)       ONES GraphQL / REST             │
│   (项目元数据/工作项创建)          (项目列表/工作项类型)            │
```

### 1.2 ONES 插件平台集成方式

DCP 评审中心通过以下 ONES 插件能力与宿主平台集成：

| 集成能力 | 类型 | 说明 |
|---------|------|------|
| **External API** | `type: external` | 后端通过 `apis` 段注册 27 个 REST 端点，前端通过 `/project/api/project/team/{teamUUID}` 前缀路由访问，需携带 `Ones-Plugin-Id` 请求头 |
| **Entity Storage** | `storage.entities` | 声明 15 个自定义实体表，使用 `storage.entity(name)` 获取句柄，支持 `.get()/.set()/.delete()/.query()` 操作 |
| **SidebarMenu** | `abilityType: SidebarMenu` | 注册侧边栏菜单入口 `dcp-sidebar`，包含「模板配置」和「评审总览」两个子页面 |
| **ProjectCustomComponent** | `abilityType: ProjectCustomComponent` | 注册项目自定义组件 `dcp-review-tab`，可作为项目页签嵌入项目内；声明 `copyPluginData` 钩子（当前返回不复制） |
| **OPFetch** | `@ones-op/fetch` | 后端通过 OPFetch 调用 ONES 内部 API（项目元数据解析、工作项创建、工作项类型查询），自动携带团队鉴权 |
| **Notify** | `@ones-op/node-ability` | 通过 Notify 能力发送通知，支持邮件(Email)、微信(WeChat)、钉钉(DingDing)、飞书(Lark)、有道(YouDu) 五种渠道 |
| **微前端容器** | `__ONES_MF_ENV__` | 前端从 ONES 微前端环境变量获取 teamUUID、pluginID、instanceId 等上下文 |

### 1.3 运行模式

- `service.mode: org` — 团队级插件，数据按团队隔离
- `service.scope: 1` — 组织级作用域
- `service.language: nodejs` — Node.js 运行时
- 最低系统版本 `3.11.39`，当前 host 版本 `1.11.32`

## 2. 模块划分

### 2.1 模块总览

| 模块 ID | 模块类型 | 入口文件 | 职责 |
|---------|---------|---------|------|
| `dcp-review-tab` | `ones:project:component:new` | `./modules/dcp-review-tab/index.tsx` | 项目内 DCP 评审工作台 |
| `dcp-reviewer-workspace` | `ones:workspace:new` | `modules/dcp-reviewer-workspace/index.html` | 评审人工作台 |
| `dcp-sidebar` | `ones:app:new` | — | 侧边栏应用容器 |
| `dcp-template-config` | 子模块(dcp-sidebar) | `modules/dcp-sidebar/dcp-template-config/index.html` | 模板配置页 |
| `dcp-review-overview` | 子模块(dcp-sidebar) | `modules/dcp-sidebar/dcp-review-overview/index.html` | 评审总览页 |

### 2.2 各模块详细说明

#### 2.2.1 dcp-review-tab（项目内评审组件）

- **类型**：`ones:project:component:new`（项目自定义组件）
- **权限**：无单独权限点，跟随项目访问权限
- **职责**：在项目页面内以页签形式展示 DCP 评审工作台，是创建和管理评审单的主入口
- **核心功能**：
  - 评审列表展示（分页，按视口高度自适应每页 5~15 条）
  - 创建评审单（选择 DCP 阶段、填写标题、会议时间）
  - 评审详情（材料/指标/评审人/意见/关联工作项/决议/审计 六大 Tab）
  - 材料上传/下载/预览
  - 评审人维护、发起评审
  - 评审单直链复制（构建指向 dcp-review-overview 的深链接）
- **前端上下文获取**：从 `window.parent.location.hash` 解析 `projectUUID/projectKey/componentUuid/viewUuid`，通过 exchange API + stamps API 补查项目名称

#### 2.2.2 dcp-reviewer-workspace（评审人工作台）

- **类型**：`ones:workspace:new`
- **职责**：评审人的个人工作台，展示待办/已办评审，支持提交评审意见
- **核心功能**：
  - 待评审 / 已评审 Tab 切换
  - 评审详情查看（材料、指标、其他评审人意见、checklist）
  - 提交评审意见（结论 pass/conditional_pass/fail、风险等级、意见摘要）
  - 深链接直达（`?review_uuid=xxx`）
  - checklist 勾选（按角色匹配）
  - 创建工作项并自动关联

#### 2.2.3 dcp-review-overview（团队评审总览）

- **类型**：dcp-sidebar 子模块，权限点 `dcp_view_review`
- **职责**：团队级评审全景视图，汇总所有项目的评审数据
- **核心功能**：
  - 统计卡片（总评审数、评审中、已完成、关联工作项数）
  - 评审列表（可排序、可按项目折叠分组）
  - 评审详情（复用 dcp-review-tab 的 ReviewDetail 组件）
  - 项目名称前端兜底补查（hydrateProjectNames）
  - 深链接直达 + 编号链接复制

#### 2.2.4 dcp-template-config / dcp-config-page（模板配置）

- **类型**：dcp-sidebar 子模块，权限点 `dcp_admin`
- **职责**：全局评审模板配置，管理员维护阶段/材料/指标/角色/checklist/通知设置
- **配置项**：
  - **阶段模板**：DCP1~DCP4 阶段、决议选项、决议模板
  - **材料模板**：材料名称、适用阶段、是否必填、负责角色
  - **指标模板**：指标名称、适用阶段、单位、阈值类型（达标/高于/低于预警）、黄/红阈值
  - **评审角色**：角色名称、是否必投(must_vote)、是否有否决权(has_veto)、排序
  - **Checklist 模板**：检查项文本、适用阶段、负责角色
  - **通知设置**：启用开关、三个通知触发点（发起/全部提交/发布决议）、五种通知渠道
- **编辑机制**：编辑态开关控制，保存时后端先删除旧记录再全量写入（replace 策略）

#### 2.2.5 dcp-sidebar（侧边栏容器）

- **类型**：`ones:app:new` + `abilityType: SidebarMenu`
- **职责**：通过 SidebarMenu 能力注册左侧导航入口，承载 dcp-template-config 和 dcp-review-overview 两个子页面
- **子页面入口**：各自独立 HTML 文件，内部 `ReactDOM.render` 挂载对应 React 组件

## 3. 后端 API 设计

后端共注册 27 个 external API 端点（`plugin.yaml` 的 `apis` 段），全部定义在 `backend/src/index.ts` 中。所有端点路由前缀为 `/project/api/project/team/:teamUUID`，需携带 `Ones-Plugin-Id` 请求头。

### 3.1 配置管理 API

| 方法 | 路径 | Handler 函数 | 功能说明 |
|------|------|-------------|---------|
| GET | `/dcp/config` | `getPluginConfig` / 别名 `getDcpConfig` | 获取全局配置：阶段/材料/指标/角色/checklist 模板 + 通知配置 + 默认决议模板 |
| POST | `/dcp/config` | `savePluginConfig` | 保存全局配置，采用「先删旧记录再全量写入」的 replace 策略 |

### 3.2 评审单管理 API

| 方法 | 路径 | Handler 函数 | 功能说明 |
|------|------|-------------|---------|
| POST | `/dcp/review` | `createReview` | 创建评审单，自动带出适用阶段的材料模板和指标模板，生成唯一编号 `{项目标识}{YYYYMMDD}{两位序号}` |
| GET | `/dcp/review/:review_uuid` | `getReviewDetail` | 获取评审单聚合详情（评审单+材料+指标+评审人+关联工作项+决议+补充说明+checklist） |
| GET | `/dcp/reviews/by-project/:project_uuid` | `listReviewsByProject` | 按项目列出评审单，含评审人完成情况/材料进度/关联工作项数 |
| GET | `/dcp/reviews` | `getDcpReviews` | 别名路由，带 `project_uuid` 参数时转发到 `listReviewsByProject`，否则转发到 `listTeamReviews` |
| GET | `/dcp/reviews/my` | `listMyReviews` | 按评审人 UUID 筛选待办/已办评审，从 URL query 解析 `reviewer_uuid` |
| GET | `/dcp/reviews/team` | `listTeamReviews` | 团队全部评审，含统计数据（总数/评审中/已完成/关联工作项数），批量解析项目元数据 |
| POST | `/dcp/review/:review_uuid/start` | `startReview` | 发起评审（draft→reviewing），含三重校验：必投角色已指定、必填材料已上传、关键指标无红线 |
| DELETE | `/dcp/review/:review_uuid` | `deleteReview` | 删除评审单（仅草稿状态、仅创建者），级联删除所有关联子实体 |

### 3.3 材料管理 API

| 方法 | 路径 | Handler 函数 | 功能说明 |
|------|------|-------------|---------|
| POST | `/dcp/review/:review_uuid/material-status` | `updateMaterialStatus` | 更新材料提交状态（pending/submitted/approved/rejected）和备注 |
| POST | `/dcp/review/:review_uuid/material-upload` | `uploadMaterialFile` | 上传材料文件，记录 object_key 到 file_data 字段 |
| POST | `/dcp/review/:review_uuid/material-remove` | `removeMaterialFile` | 清除材料文件（仅草稿状态可操作） |
| GET | `/dcp/review/:review_uuid/material/:template_id/upload-url` | `getMaterialUploadUrl` | 获取对象存储预签名上传 URL |
| GET | `/dcp/review/:review_uuid/material/:template_id/download-url` | `getMaterialDownloadUrl` | 获取对象存储预签名下载 URL |
| GET | `/dcp/review/:review_uuid/material/:template_id/preview` | `getMaterialPreview` | 后端代理获取文件内容，base64 返回，绕过 Content-Disposition: attachment |

### 3.4 指标管理 API

| 方法 | 路径 | Handler 函数 | 功能说明 |
|------|------|-------------|---------|
| POST | `/dcp/review/:review_uuid/indicators` | `updateIndicators` | 批量更新指标值，自动计算风险颜色（green/yellow/red，基于阈值类型和黄/红阈值） |

### 3.5 评审人管理 API

| 方法 | 路径 | Handler 函数 | 功能说明 |
|------|------|-------------|---------|
| POST | `/dcp/review/:review_uuid/reviewers` | `updateReviewers` | 维护评审人列表（仅草稿状态），校验角色合法性，按 sort_order 排序，同步写入 `reviewers_json` 快照 |
| POST | `/dcp/review/:review_uuid/opinion` | `submitOpinion` | 提交评审意见（仅 reviewing 状态），禁止重复提交，同步更新快照，全部提交后通知 Chair |

### 3.6 关联工作项 API

| 方法 | 路径 | Handler 函数 | 功能说明 |
|------|------|-------------|---------|
| POST | `/dcp/review/:review_uuid/link-issue` | `linkIssue` | 关联已有工作项到评审单，防重复 |
| GET | `/dcp/review/:review_uuid/linked-issues` | `getLinkedIssues` | 获取评审单关联的工作项列表 |
| POST | `/dcp/review/:review_uuid/create-issue` | `createIssue` | 创建工作项并自动关联评审单，多路径级联尝试内部 API（tasks/add3 → tasks → items → task），失败返回 fallback URL |
| POST | `/dcp/project/:project_uuid/issue-types` | `listIssueTypes` | 获取项目工作项类型列表，优先 GraphQL，降级 REST 多路径级联，最后降级 Open API v2/v1 |

### 3.7 决议管理 API

| 方法 | 路径 | Handler 函数 | 功能说明 |
|------|------|-------------|---------|
| POST | `/dcp/review/:review_uuid/generate-resolution` | `generateResolution` | 兼容旧名，直接转发到 `publishResolution` |
| POST | `/dcp/review/:review_uuid/publish-resolution` | `publishResolution` | 发布决议（仅 Chair、全部 must_vote 角色提交后、不可覆盖）。校验通过决议需≥3位非 Chair 评审人投通过/有条件通过。生成快照编号 `DCP-RES-{时间戳36进制}`，冻结全员投票快照到 `based_on_votes` |

### 3.8 补充说明 API

| 方法 | 路径 | Handler 函数 | 功能说明 |
|------|------|-------------|---------|
| POST | `/dcp/review/:review_uuid/supplement` | `addSupplement` | 添加补充说明(supplement)或纠偏说明(rectification) |

### 3.9 审计日志 API

| 方法 | 路径 | Handler 函数 | 功能说明 |
|------|------|-------------|---------|
| GET | `/dcp/review/:review_uuid/audit-log` | `getAuditLog` | 获取评审单操作审计日志，按时间倒序排列 |

### 3.10 Checklist API

| 方法 | 路径 | Handler 函数 | 功能说明 |
|------|------|-------------|---------|
| POST | `/dcp/review/:review_uuid/checklist` | `checkChecklist` | 勾选/取消 checklist 项（仅 reviewing 状态），权限校验：操作人须为对应角色的评审人且非 Chair |

### 3.11 诊断 API

| 方法 | 路径 | Handler 函数 | 功能说明 |
|------|------|-------------|---------|
| GET | `/dcp/debug/reviewer-storage/:review_uuid` | `debugReviewerStorage` | 诊断接口：直接读实体 key 验证评审人存储行为，含 qAll 全量 + 按 key 直读 |
| GET | `/dcp/debug/my-reviews` | `debugMyReviews` | 诊断接口：dump listMyReviews 原始数据，排查评审人匹配问题 |

## 4. 实体存储模型

系统声明 15 个自定义实体（`plugin.yaml` 的 `storage.entities` 段），分为「配置模板」「评审业务」「审计辅助」三类。所有 text 类型字段有 length 上限约束，integer 字段以 Unix 时间戳存储时间。

### 4.1 配置模板实体（5 个）

#### dcp_base_config — 基础配置键值表

| 属性 | 类型 | 长度 | 默认值 | 用途 |
|------|------|------|--------|------|
| `key` | text | 64 | '' | 配置键名（如 default_resolution_template、notify_config、review_seq_xxx） |
| `value` | text | 32768 | '' | 配置值（notify_config 存 JSON 字符串） |

#### dcp_phase_template — 阶段模板

| 属性 | 类型 | 长度 | 默认值 | 用途 |
|------|------|------|--------|------|
| `phase_code` | text | 16 | '' | 阶段代码（如 DCP1、DCP2） |
| `phase_name` | text | 64 | '' | 阶段名称 |
| `resolution_options` | text | 4096 | '[]' | 决议选项 JSON 数组 |
| `resolution_template` | text | 8192 | '' | 决议模板文本 |
| `sort_order` | integer | 8 | 0 | 排序序号 |

#### dcp_material_template — 材料模板

| 属性 | 类型 | 长度 | 默认值 | 用途 |
|------|------|------|--------|------|
| `material_name` | text | 128 | '' | 材料名称 |
| `applicable_phases` | text | 512 | '[]' | 适用阶段 JSON 数组 |
| `required` | boolean | — | false | 是否必填 |
| `responsible_role` | text | 64 | '' | 负责角色 |
| `sort_order` | integer | 8 | 0 | 排序序号 |

#### dcp_indicator_template — 指标模板

| 属性 | 类型 | 长度 | 默认值 | 用途 |
|------|------|------|--------|------|
| `indicator_name` | text | 128 | '' | 指标名称 |
| `applicable_phases` | text | 512 | '[]' | 适用阶段 JSON 数组 |
| `unit` | text | 16 | '' | 单位 |
| `threshold_type` | text | 32 | '达标即通过' | 阈值类型（达标即通过/高于阈值预警/低于阈值预警） |
| `yellow_threshold` | float | 16 | 0 | 黄线阈值 |
| `red_threshold` | float | 16 | 0 | 红线阈值 |
| `sort_order` | integer | 8 | 0 | 排序序号 |

#### dcp_reviewer_role — 评审角色模板

| 属性 | 类型 | 长度 | 默认值 | 用途 |
|------|------|------|--------|------|
| `role_name` | text | 64 | '' | 角色名称（如 Chair、研发VP） |
| `must_vote` | boolean | — | true | 是否必须投票 |
| `has_veto` | boolean | — | false | 是否有否决权 |
| `sort_order` | integer | 8 | 0 | 排序序号 |

> 注：`Enable()` 生命周期钩子在首次启动时初始化 6 个默认角色（Chair、研发VP、市场VP、财务代表、质量代表、供应链代表）。

### 4.2 评审业务实体（6 个）

#### dcp_review — 评审单主表

| 属性 | 类型 | 长度 | 默认值 | 用途 |
|------|------|------|--------|------|
| `review_uuid` | text | 64 | '' | 评审单唯一标识（业务主键） |
| `project_uuid` | text | 64 | '' | 所属项目标识（可能为 key 或真实 UUID） |
| `phase_code` | text | 16 | '' | DCP 阶段代码 |
| `review_title` | text | 256 | '' | 评审标题 |
| `meeting_time` | integer | 16 | 0 | 会议时间戳 |
| `status` | text | 32 | 'draft' | 状态（draft/reviewing/completed/rejected） |
| `creator_uuid` | text | 64 | '' | 创建者 UUID |

## 5. 关键技术实现

### 5.1 实体存储与分页

#### qAll cursor 分页机制

ONES Entity Storage 的 `getMany()` 方法硬限制每页最多返回 200 条记录。系统封装了通用 `qAll()` 函数实现自动游标分页：

```typescript
async function qAll(e: any, filter?: (v: any) => boolean) {
  const allItems: any[] = []
  let cursor: string | null = null
  let safety = 0
  while (safety < 100) {           // 安全阀：最多 100 页 = 20000 条
    safety++
    const q = e.query().limit(200) // 每页硬限 200
    if (cursor) q.cursor(cursor)
    const result = await q.getMany()
    if (!result || !Array.isArray(result.data)) break
    for (const d of result.data) {
      allItems.push({ _key: d.key, ...(d.value || {}) })
    }
    const pi = result.page_info
    if (pi && pi.has_more && pi.end_cursor) {
      cursor = pi.end_cursor       // 翻页
    } else {
      break
    }
  }
  return filter ? allItems.filter(filter) : allItems
}
```

- **分页参数**：`limit(200)` + `cursor(end_cursor)` + `page_info.has_more`
- **安全阀**：最多循环 100 次（20000 条），防止无限分页
- **内存过滤**：由于 entity storage 不支持服务端条件过滤，`qAll` 先全量拉取再在内存中按 `filter` 函数过滤
- **性能影响**：每次查询评审详情需 6~8 次 `qAll` 并发调用（材料、指标、评审人、关联工作项、决议、补充说明 + 阶段/材料/指标模板），大量评审单时 `listTeamReviews` 会对每条评审单执行多次 `qAll`

#### 父实体 JSON 快照兜底

由于 ONES Entity Storage 存在「子实体在某些请求上下文中 qAll 不可见」的已知问题，系统采用双写策略：

1. **reviewers_json 快照**：`updateReviewers` 写入 `dcp_review_reviewer` 实体的同时，将评审人列表 JSON 序列化写入 `dcp_review.reviewers_json` 字段（32KB）。读取时优先读快照，快照为空再降级读实体：
   ```typescript
   const snapReviewers = jsonArr((rv as any).reviewers_json || '[]')
   const reviewers = snapReviewers.length > 0
     ? snapReviewers
     : await qAll(rvReviewer, (v: any) => v.review_uuid === rid)
   ```
2. **checklist_json 快照**：`startReview` 时从 `dcp_checklist_item` 模板初始化 checklist 到 `dcp_review.checklist_json`，后续 `checkChecklist` 直接操作快照字段，不再依赖子实体
3. **submitOpinion 同步快照**：每次提交评审意见后，同步更新 `reviewers_json` 中对应评审人的 conclusion/risk_level/opinion_summary/submitted_at，确保快照路径读到最新数据

### 5.2 文件附件管理

#### 存储方案

材料文件采用 **ONES Object Storage + 实体引用** 双层方案：

| 层级 | 存储位置 | 内容 |
|------|---------|------|
| 对象存储 | ONES Object Storage | 文件二进制内容，object_key 格式 `dcp_files-{rid}-{tid}-{timestamp}` |
| 实体引用 | `dcp_review_material.file_data` | 存储 object_key（非 base64 内容），10MB 上限 |
| 元数据 | `dcp_review_material.file_name/file_size` | 文件名和大小 |

#### 上传/下载/预览 API 设计

```
上传流程：
  前端 → GET /material/:tid/upload-url → 获取预签名 URL + fields + object_key
       → PUT 预签名 URL（直传对象存储）
       → POST /material-upload → 将 object_key 写入 file_data 字段

下载流程：
  前端 → GET /material/:tid/download-url → 获取预签名下载 URL
       → 浏览器直接访问 URL 下载

预览流程（绕过 Content-Disposition: attachment）：
  前端 → GET /material/:tid/preview → 后端代理获取文件内容
       → OPFetch 下载 → Buffer → base64 → 返回 { content, mime, file_name }
       → 前端用 data URI 或 Blob 渲染
```

**预览 MIME 推断**：后端根据文件扩展名映射 MIME 类型（pdf/png/jpg/docx/xlsx/pptx/zip 等 15 种），未知类型返回 `application/octet-stream`。

**对象存储 key 约束**：ONES 对象存储不支持 `/` 路径分隔符，object_key 使用 `-` 连接。

### 5.4 评审流程状态机

#### 状态流转

```
┌────────┐  startReview    ┌───────────┐ publishResolution(pass/conditional_pass) ┌───────────┐
│ draft  │ ──────────────→ │ reviewing │ ──────────────────────────────────────→ │ completed │
│        │                 │           │                                          └───────────┘
└───┬────┘                 └─────┬─────┘ publishResolution(reject)
    │                            │
    │ deleteReview               ▼
    ▼                      ┌──────────┐
┌──────────┐               │ rejected │
│ (deleted)│               └──────────┘
└──────────┘
```

#### 各状态操作约束

| 操作 | draft | reviewing | completed/rejected |
|------|-------|-----------|-------------------|
| 删除评审单 | ✅ 仅创建者 | ❌ | ❌ |
| 修改评审人 | ✅ | ❌ | ❌ |
| 上传/清除材料文件 | ✅（清除仅草稿） | ✅（仅上传） | ❌ |
| 修改指标 | ✅ | ✅ | ❌ |
| 修改材料状态 | ✅ | ✅ | ❌ |
| 提交评审意见 | ❌ | ✅ | ❌ |
| Checklist 勾选 | ❌ | ✅ | ❌ |
| 发布决议 | ❌ | ✅ 仅 Chair | ❌ |
| 添加补充说明 | ✅ | ✅ | ✅ |

#### startReview 三重校验

发起评审前执行三项前置校验，任一失败即拒绝：

1. **必投角色校验**：所有 `must_vote=true` 的角色模板必须已指定评审人
2. **必填材料校验**：所有 `required=true` 的材料模板必须已上传文件（`file_data` 非空）
3. **红线指标校验**：所有指标的 `risk_color` 不能为 `red`（超出红线阈值）

#### publishResolution 决议规则

- **操作人校验**：必须是 `Chair` 角色评审人（`chair.reviewer_uuid === publisher_uuid`）
- **提交完整性校验**：所有 `must_vote` 角色的评审人必须已提交意见（`submitted_at > 0`）
- **多数决议规则**：决议为 `pass` 时，非 Chair 评审人中至少 3 人投 `pass` 或 `conditional_pass`
- **不可覆盖机制**：同一评审单只能发布一次决议，重复发布返回 409 Conflict

#### 决议快照不可覆盖

发布决议时冻结全员投票快照到 `dcp_resolution.based_on_votes`（65KB JSON），包含每位评审人的 UUID、角色、结论、风险等级、意见摘要、提交时间。快照生成后不可修改，确保审计追溯的完整性。

### 5.5 审计日志

#### writeAudit 机制

系统通过统一的 `writeAudit()` 函数记录所有关键操作：

```typescript
async function writeAudit(
  rvUuid: string,      // 评审单 UUID
  op: string,          // 操作人 UUID（前端传入 operator_uuid）
  action: string,      // 操作名称
  target: string,      // 操作目标
  detail: string,      // 操作详情
  result = 'success'   // 操作结果
) {
  const k = `${rvUuid}_${Date.now()}_${Math.random().toString(36).slice(2, 6)}`
  await auditLog.set(k, {
    review_uuid: rvUuid, timestamp: Date.now(),
    operator_uuid: op || '', action, target, detail, result,
  })
}
```

**设计要点**：
- **operator_uuid 前端传入**：由于 ONES external API handler 不自动注入当前用户身份，操作人 UUID 由前端请求体 `operator_uuid` 字段显式传入
- **key 唯一性**：`{review_uuid}_{timestamp}_{random4}` 确保并发操作不冲突
- **覆盖操作**：创建评审、启动评审、上传/删除材料、更新评审人、提交评审意见、关联/创建工作项、发布决议、添加补充说明均写审计日志

### 5.7 前端项目上下文获取

#### window.parent.location.hash 解析

`dcp-review-tab` 运行在项目页面的 iframe 内，无法直接获取 ONES 页面上下文。系统通过解析父窗口 URL hash 提取项目信息：

```typescript
// ONES 项目页面 URL 格式：
// #/team/{teamUUID}/project/{projectKey}/component/{componentUuid}/view/{viewUuid}

// 1. 提取 projectUUID（优先 query 参数，其次 hash 路径）
puid = new URLSearchParams(window.parent.location.search).get('projectUUID')
  || window.parent.location.hash.match(/\/project\/([A-Za-z0-9]+)/)?.[1]

// 2. 提取 projectKey、componentUuid、viewUuid
const ck = hash.match(/\/project\/([A-Za-z0-9]+)/)  // projectKey
const cm = hash.match(/\/component\/([A-Za-z0-9]+)/) // componentUuid
const vm = hash.match(/\/view\/([A-Za-z0-9]+)/)      // viewUuid
```

#### exchange API + stamps API 补查项目名称

projectKey 可能是项目标识符（如 `LIXM`）而非真实 UUID，需通过两级 API 解析：

```
Step 1: exchange API → 真实 UUID + 准确 identifier
  GET /project/api/ones-project/team/{tu}/projects/exchange/{projectKey}
  → { identifier: "LIXM", project_uuid: "真实UUID" }

Step 2: GraphQL → 项目名称（优先）
  POST /project/api/project/team/{tu}/items/graphql
  → { uuid, identifier, name, key }

Step 3: stamps API → 项目名称（兜底）
  POST /project/api/project/team/{tu}/project/{realUUID}/stamps/data?t=project
  → { project: { projects: [{ name, identifier, uuid }] } }
```

**前端兜底**：`dcp-team-overview` 和 `dcp-reviewer-workspace` 模块在后端解析失败时，通过浏览器侧 `fetch` 补查项目名称（`hydrateProjectNames`），确保总览页面项目名称可读。

#### teamUUID 获取（多级降级）

前端 `getTeamUUID()` 按以下优先级获取团队 UUID：
1. `__ONES_MF_ENV__.request.headers['Ones-Check-Id']`（微前端请求头）
2. `__ONES_MF_ENV__.teamUUID` / `team_uuid` / `contextStore.teamInfo.uuid`
3. `__ONES_STORE__.getState().contextStore.teamInfo.uuid`
4. 插件资源路径正则 `/plugin/{org}/{team}/{app}/`
5. 父窗口 hash `/team/{uuid}/`

## 6. 关键约束与已知限制

### 6.1 ONES 实体属性只能新增不能删除

ONES Entity Storage 的属性注册为只增模式：一旦在 `plugin.yaml` 中声明并部署，实体属性**只能新增、不能删除或重命名**。尝试删除属性会触发 `EntityAttrDeletedForbidden` 错误。

- **影响**：废弃字段仍保留在实体定义中，需通过代码层忽略
- **应对**：版本迭代时只追加新属性，旧字段保持兼容（如 `resolution_result` / `resolution_body` / `generated_by` / `generated_at` 等兼容旧字段保留）
- **Upgrade 迁移**：`Upgrade()` 钩子仅记录日志，实际属性迁移在 `plugin.yaml` 声明后由平台自动执行

### 6.2 字段容量限制

| 字段类型 | 上限 | 影响 |
|---------|------|------|
| text（常规） | ~64KB（65535 字节） | `based_on_votes` 冻结投票快照接近上限，大量评审人时需注意 |
| text（file_data） | 10MB（10485760 字节） | 材料文件 object_key 存储在此字段，实际文件内容在对象存储 |
| text（reviewers_json / checklist_json） | 32KB | 评审人/checklist 快照，约可存 50~100 条记录 |
| getMany() 每页 | 200 条 | 必须使用 cursor 分页，`qAll` 封装自动处理 |

### 6.3 handler 持久化 set() 限制

单个 API handler 的一次请求内，`storage.entity.set()` 调用次数存在限制（≤2 条/请求的写入配额在某些 ONES 版本中存在）。系统通过以下方式应对：

- **批量写入拆分**：`updateReviewers` 先删除旧记录（逐条 delete），再逐条 set 新记录
- **配置保存全量替换**：`savePluginConfig` 对每个模板表执行「先删旧再写新」的 replace 策略
- **双写优化**：`updateReviewers` 同时写子实体和父实体快照，减少后续 qAll 依赖

### 6.4 getMany() 硬限分页

`entity.query().getMany()` 每页硬限 200 条，且不支持服务端条件过滤：

- **无 WHERE 子句**：所有条件过滤在内存中执行（`qAll` 的 filter 参数）
- **性能瓶颈**：`listTeamReviews` 对每条评审单执行 3~4 次 qAll（评审人、关联工作项、材料），N 条评审单 = O(N) 次 qAll 调用
- **安全阀**：`qAll` 最多循环 100 页（20000 条），超出则截断

### 6.5 external API handler req 对象限制

ONES external API 的 `req` 对象**不暴露标准 Express 的 `params`/`query`/`path` 属性**：

- **路径参数**：`req.params` 可能为空，需通过 `getParam()` 函数从 `req.url` 正则提取
- **查询参数**：`req.query` 可能为空，需从 `req.url` 手动解析 query string（如 `?reviewer_uuid=xxx`）
- **应对**：系统封装 `getParam(req, name)` 函数，按命名路径段 → 特殊路径模式 → 末尾段的优先级提取参数

```typescript
function getParam(req: any, name: string): string {
  if (req.params?.[name]) return req.params[name]
  const url = (req.url || '').split('?')[0]
  // 匹配 /{name}/{value}
  const named = url.match(new RegExp(`/${name}/([^/]+)`))
  if (named) return named[1]
  // 特殊路径模式兜底...
}
```

### 6.6 Ones-Plugin-Id 头必须设置

所有前端 API 请求必须携带 `Ones-Plugin-Id` 请求头，值为插件 app_id（`709xehle`，无 `dev_` 前缀）：

- **API 请求**：`Ones-Plugin-Id: 709xehle`（external API 路由匹配）
- **权限检查**：`Ones-Plugin-Id: built_in_apis`（batch_check 端点专用）
- **缺失后果**：请求返回 404 或路由不匹配

### 6.7 其他已知限制

- **跨域限制**：`window.parent.location` 在跨域 iframe 中不可访问，需从 URL hash / script src 降级提取上下文
- **plugin/list 不返回 org 模式插件**：`mode:org` + `ProjectCustomComponent` 插件的 instance_id 无法通过 `plugin/list` 获取，需维护已知映射
- **对象存储路径分隔符**：ONES 对象存储 object_key 不支持 `/`，需用 `-` 替代
- **通知非阻塞**：Notify 调用失败仅记日志，不重试，可能导致通知丢失
- **决议不可覆盖**：一旦发布决议，评审单状态锁定，无法重新评审（需新建评审单）
